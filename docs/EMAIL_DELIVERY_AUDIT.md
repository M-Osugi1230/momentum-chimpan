# Email SMTP Acceptance Audit

Last updated: 2026-07-13

## Purpose

The daily report now records whether email was:

- accepted by the configured SMTP server;
- skipped because required secrets were missing;
- attempted and failed with an exception.

This is an SMTP acceptance audit. It does not prove final inbox delivery, spam-folder placement, opening, or reading.

## Daily receipt

File inside the daily artifact:

`output/email_delivery_receipt.json`

The receipt is generated by `email_delivery.py` through the `daily_runner.py` display and delivery wrapper.

Statuses:

| Status | Meaning |
|---|---|
| `SMTP_ACCEPTED` | The existing email function returned without an SMTP exception while all required credentials were configured |
| `SKIPPED_SECRETS_MISSING` | Sender, recipient, or app-password configuration was incomplete |
| `FAILED` | The email function raised an exception; the original exception is re-raised so the daily report step still fails |

## Privacy boundary

The receipt never stores:

- sender email address;
- recipient email address;
- Gmail app password;
- message body;
- raw exception message.

It stores only:

- SHA-256 of the normalized sender identity;
- SHA-256 of the sorted recipient set;
- recipient count;
- SHA-256 of the subject;
- exception class when failed;
- report date and GitHub run identifiers;
- SMTP host and port;
- receipt and full-status integrity hashes.

This allows operational correlation without exposing contact or credential data in Actions artifacts.

## Claims and limitations

The receipt fixes:

- `inbox_delivery_claimed: false`;
- `error_message_stored: false`;
- `automatic_retry: false`;
- `automatic_strategy_change: false`;
- `production_state_mutations: []`.

`SMTP_ACCEPTED` means the application observed no SMTP rejection. Gmail or another downstream system may still delay, filter, bounce, or reject the message later.

## Signed audit ledger

Files:

- `research/operations/email_delivery_audit.csv`;
- `research/operations/email_delivery_audit_status.json`.

The `Email Delivery Audit` workflow runs after every completed `Daily Momentum Report` and downloads the exact upstream artifact by run ID.

The ledger natural key is `workflow_run_id`. Reprocessing the same run replaces the row rather than duplicating it.

Each row records:

- upstream workflow result and event;
- report date;
- receipt presence and signature validity;
- SMTP status;
- attempted and accepted flags;
- configuration completeness;
- recipient count;
- failure class;
- receipt and artifact fingerprints;
- audit validity and failure reason.

A missing or unreadable receipt is retained as an audit failure instead of being silently skipped.

## Publisher permissions

The validation job is read-only.

Only the isolated publish job has `contents: write`, and it stages only:

- `research/operations/email_delivery_audit.csv`;
- `research/operations/email_delivery_audit_status.json`.

The workflow does not receive email secrets and cannot send mail.

It shares the production-audit concurrency group so the operational publishers do not push conflicting audit commits simultaneously.

## Monthly metrics

The monthly review now reports:

- receipt row count;
- valid-receipt rate;
- scheduled receipt count;
- scheduled email attempt count;
- scheduled SMTP accepted count;
- scheduled SMTP acceptance rate;
- scheduled secret-configuration skips;
- scheduled send failures;
- inbox-delivery observability, always false.

The one-row CSV includes these fields, and the Markdown report separates SMTP acceptance from final inbox delivery.

## Failure behavior

### Missing secrets

The original email function keeps its existing skip behavior. The receipt makes that skip observable.

A scheduled skip reduces the monthly SMTP acceptance rate and is shown separately from a send failure.

### SMTP exception

The receipt is written first with status `FAILED`, then the original exception is re-raised. The daily workflow therefore remains failed and the failure notification path remains active.

### Missing receipt

The email audit records an invalid/missing receipt. It does not infer success from the presence of the workbook or from overall workflow status.

## Non-goals

This subsystem does not:

- retry messages automatically;
- inspect Gmail inbox or Sent Mail;
- request read receipts;
- track opens or clicks;
- expose email addresses;
- change report content;
- change strategy, score, priority, paper execution, or production state;
- place live orders.
